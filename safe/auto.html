# Automated Smart Contract Check

All users’ approved contracts are scanned by an algorithm for automated security vulnerabilities detection designed by DefiYield. The algorithm is able to inform users about the following SC metrics:

  • Is the analyzed SC code verified by Etherscan or not? This reflects whether the contract bytecode matches with that on the blockchain.
  • Owner type. Here users can see if the smart contract is owned by an EOA, a SC or it doesn’t have any owner. If the owner is an EOA, it can interact with the smart contract and call underlying functions at his own discretion.
  • Major EOA-holders. Users can see if there are EOAs holding large shares of the project token. (One private owner has > 15% of the token total supply.) This aspect doesn't refer to code security problems, however, users should take into consideration that the token can be dumped by the major holder/s anytime.
  • Abandoned. The project is considered abandoned if no transactions were broadcasted through the smart contract of its token in the last 30 days
  • Migration. If migration functionality is present in the token smart contract, user funds are endangered.
  • Pause. If the smart contract can be paused, users don’t have constant access to their funds and ability to move them and, thus, face a higher risk of money loss.
  • Infinite minting. If the project token is mintable:
    • Tokens can be minted to an EOA to perform a rug pull
    • Unlimited token supply can cause inflation of the token The algorithm also indicates who can mint tokens: a regular EOA, a smart contract or no minter is set.
  • New. The contract was created less than 7 days ago and has not been battle-tested yet: code vulnerabilities can be revealed in the future
  • Proxy. As code of already deployed smart contracts can’t be upgraded, some projects choose proxy contract pattern. A proxy contract stores variables. Receiving message calls, it redirects them to a contract containing the logic. This architecture allows projects to bind newly deployed contracts with new logic to already existing dependencies as if the main logic gets upgraded.
    This exposes users to the risk of losing funds as malicious logic can be introduced with a newly deployed contract.
  • Mixers used. A mixer can be used by the dev team to pay for deployment of the project contracts. This trick is often applied by scammy projects to be untraceable after performing a rug pull.
